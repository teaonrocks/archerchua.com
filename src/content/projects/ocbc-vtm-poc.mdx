When I signed up for the OCBC Ignite Innovation Challenge 2025, I had a simple idea: what if banking could be as natural as talking to a teller, but available 24/7? I already had experience with local AI and React, but I had never touched WebRTC before. The challenge of building real-time video, audio, and screen sharing on a tight deadline was both intimidating and irresistible. The result was a Virtual Teller Machine prototype that became my crash course in building production-grade AI systems and mastering protocols I barely understood a week before.

## The Solution

**OCBC VTM POC** is a full-stack smart banking kiosk that turns voice requests into completed transactions. From the customer's perspective, they speak their request aloud, the kiosk captures and transcribes their voice in the browser using Whisper, a local LLM (llama3.2:3b via Ollama) classifies that text into a structured banking intent, and the UI responds with guided workflows for remittances, card replacement, loan enquiries and more, all presented as dynamic cards. When self-service is not enough, a single click escalates the session to a live agent over WebRTC, complete with screen sharing, video, audio and a real-time annotation overlay so the agent can literally point at the problem on-screen.

Under the hood, a monorepo architecture stitches together five moving parts: a TanStack Start kiosk UI, a Python ASR microservice, a WebSocket signalling server, a shared WebRTC client package, and a standalone Live Agent dashboard.

## Design Rationale

### Why Local AI?

**Privacy by default**: Running Whisper in the browser and Ollama on-device means customer voice data never leaves the machine. For a bank, this isn't a feature; it's a requirement. It also eliminates API costs and latency spikes.

**Offline resilience**: The kiosk can function during network outages. When the LLM fails, a keyword-based fallback keeps the demo running. This taught me early that graceful degradation matters more than chasing perfect uptime.

### Why a Monorepo?

**Independent scaling**: The ASR service can run on GPU hardware while the kiosk runs on a lightweight tablet. The signalling server handles WebRTC negotiations separately. This isolation let me debug each component without bringing down the entire system.

**Hackathon velocity**: With `pnpm dev:stack`, I could spin up the entire ecosystem with one command. Judges could see the full flow immediately, which mattered more than pristine architecture.

### Tech Stack Choices

**TanStack Start**: The automatic code-splitting and server functions let me build a full-stack app without juggling separate frontend/backend repos. The type safety caught dozens of errors before runtime.

**WebRTC with annotations**: The data channel for agent drawings wasn't in any tutorial. I dug into `RTCDataChannel` APIs to sync annotations in real-time, which became a standout feature.

## Challenges & Learning

### Learning WebRTC from Zero

When I realised WebRTC would be the backbone of live agent escalation, I knew I had to learn an unfamiliar protocol fast. Instead of trying to piece everything together from scattered blog posts, I used AI as a thinking partner: I asked it to break down concepts like offers and answers, ICE candidates, and data channels in plain language, then cross-checked those explanations against MDN and official docs so I understood not just the “how” but the “why”. From there, I leaned on it to sketch minimal examples—“show me a bare-bones two-peer connection with screen sharing”—and reworked those into a shared `webrtc-client` package with React hooks that fit the monorepo.

Whenever something failed silently (no video, no errors), I could paste logs and SDP blobs into the chat and get back pointed questions about which part of the pipeline to inspect next. By the end of the hackathon, the AI wasn’t writing production code for me, but it had drastically shortened the feedback loop: I spent less time stuck on vocabulary and more time thinking about architecture, failure modes, and how WebRTC, TanStack Start, and the local AI stack all needed to interlock.

### The Security Nightmare That Consumed a Night

The biggest challenge wasn't WebRTC itself; it was **making the connection secure**. I had everything working on `localhost` with HTTP, but the moment we tried to demo over HTTPS (required for camera access), the entire system collapsed.

My teammate and I spent an entire night debugging why the WebRTC connection would initialise but never establish. The signalling server was running, the kiosk could request a live agent, but the video streams never appeared. We tried everything: different ports, different networks, different browsers.

The root cause turned out to be a cascade of security requirements I didn't fully understand yet: the kiosk UI had to run on HTTPS to use `getUserMedia`, the WebSocket signalling server needed to speak WSS, the live agent ticket bridge also had to be served over HTTPS, and all three services required valid TLS certificates—even in local development—if we wanted browsers to treat them as trustworthy. Discovering `mkcert` for locally trusted certificates was the first breakthrough, but it still took time to wire each service up with the right certificate paths, update every environment variable to use `https://` and `wss://` URLs, and chase down mixed-content warnings.

When we finally saw the agent video appear on the kiosk screen at 4 AM, the relief was immediate. The lesson was just as enduring: security isn’t something you bolt on at the end; it’s a foundational constraint that, if ignored, can quietly undermine everything else.

### The Browser is a Hostile Environment

YouTube tutorials make WebGPU look seamless. Reality looks different when Safari Technical Preview refuses to load Whisper models because of module worker restrictions and DevTools fills up with WASM memory allocation errors. That was when I really internalised that "on-device AI" is less about a single model and more about surviving every browser quirk in the wild.

The way through was an aggressive chain of feature detection and fallbacks. If WebGPU was unavailable, the app fell back to WASM; if WASM failed, it routed audio to the ASR microservice; if that service wasn't reachable, the kiosk gracefully dropped down to a typing interface. Each fallback added complexity, but they also meant the experience never completely broke.

## What I'd Do Differently

1. **WebRTC learning path**: Instead of diving into the deep end, I wish I'd built a simple video chat app first. Jumping straight into multi-stream (camera + screen share) + data channel annotations was like learning to swim in a hurricane. A weekend prototype with just peer video would have saved me days of confusion.

2. **Security-first development**: I treated TLS certificates as a deployment detail. Next time, I'll set up HTTPS/WSS on day one. The entire night we lost to security issues could have been avoided if I'd started with mkcert and required HTTPS from the beginning.

3. **Environment templates**: I hardcoded IPs across five packages. The `docs/lan-env.template` helped, but I should have used a single root `.env` file that all packages inherited. Copying variables between directories was error-prone.

4. **Feature flags**: The night before submission, I discovered a race condition in the transcription queue. I had to comment out features instead of toggling them. A simple `flags.enableWhisper` would have let me demo the fallback path cleanly.

5. **End-to-end tests**: My "testing" was speaking into my laptop at 2 AM. A single Playwright test that recorded audio, waited for transcription, and verified intent would have caught the double-transcription bug that plagued early demos.

6. **Documentation-first**: I wrote the README after the code. Next time, I'd start with the demo flowchart and API contracts. It would have prevented the signalling server from needing three rewrites.

## Impact

The prototype demonstrated that local AI is viable for banking use cases, that hybrid architecture works, and that WebRTC escalation adds trust through real-time annotation. The monorepo structure became a template for my next projects, and the WebRTC client package is now reused in other applications.

Most importantly, it transformed me from someone who used AI libraries into someone who deploys AI systems, and from a WebRTC novice into someone who can debug ICE candidate failures and TLS certificate chains.

## Conclusion

The VTM started as a hackathon experiment and evolved into a blueprint for privacy-preserving AI in regulated industries. The code has TODO comments and the ASR service lacks proper GPU memory management, but the system turns voice into action, connects strangers over WebRTC, and proves that local AI can feel seamless for end users.

Modern web development here was about orchestration, making five different technologies feel like one cohesive product. Every CORS error became a lesson about boundaries. Every WebRTC renegotiation taught me about real network conditions instead of the happy-path localhost demos.

Building this VTM showed me that the hardest part isn't the AI; it's making the AI disappear into a thoughtful interface that knows when to automate and when to connect humans. Before this project, I had never shipped anything with WebRTC. Now I have a working system with screen sharing, annotations, and live agent escalation, and a much deeper respect for the full stack between browser, network, and model.
